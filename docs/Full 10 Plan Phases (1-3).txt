PHASE 1 — FOUNDATIONS & AUDIT
Goal:
  - Understand the current Absence .NET MAUI app structure, components, styles, and configuration.
  - Identify what is already correct, what is fragile (e.g., CSS overrides), and what needs refactoring.
  - Establish a stable baseline so later phases do not break existing functionality.

Constraints:
  - Do NOT break existing functionality.
  - Do NOT remove existing CSS or styles yet.
  - Only observe, document, and annotate in this phase.
  - Any changes must be non-breaking and reversible.

----------------------------------------------------------------------
1.1 AUDIT EXISTING MAUI PROJECT STRUCTURE
----------------------------------------------------------------------

1.1.1 Locate the solution and main project
  - Open the Absence solution in VS Code / Visual Studio.
  - Identify the main .NET MAUI project (e.g., AbsenceApp, Absence.Mobile, etc.).
  - Confirm:
      - Target frameworks (e.g., net8.0-android, net8.0-ios, net8.0-windows).
      - Startup project.

1.1.2 Document the folder structure
  - In the main MAUI project, list the top-level folders:
      - /Views
      - /Pages
      - /Controls or /Components
      - /ViewModels
      - /Models
      - /Services
      - /Resources
      - /Resources/Styles
      - /Resources/Raw
      - /Resources/Fonts
      - /Resources/Images
      - /Config or /Settings (if present)
  - Create a text file in the root of the solution:
      - File: /docs/structure-phase1.txt (create /docs if it does not exist)
      - Paste the folder structure in a tree format, for example:
          - /AbsenceApp
              - /Views
              - /Pages
              - /Controls
              - /ViewModels
              - /Models
              - /Services
              - /Resources
                  - /Styles
                  - /Fonts
                  - /Images
              - /Config
  - This file is documentation only; no runtime impact.

1.1.3 Identify the app entry points
  - Open:
      - MauiProgram.cs
      - App.xaml
      - App.xaml.cs
  - In MauiProgram.cs:
      - Note how fonts, handlers, and services are registered.
      - Note any existing style dictionaries or themes being added.
  - In App.xaml:
      - Note merged resource dictionaries.
      - Note any global styles (e.g., for ContentPage, Label, Button).
  - In App.xaml.cs:
      - Note the initial MainPage.
      - Note any navigation shell or root layout (e.g., Shell, NavigationPage, custom root).

1.1.4 Identify navigation model
  - Check if the app uses:
      - Shell (AppShell.xaml / AppShell.xaml.cs)
      - NavigationPage
      - Custom root layout with manual navigation
  - Document this in /docs/structure-phase1.txt:
      - Example:
          - Navigation model: Shell-based
          - Root: AppShell.xaml
          - Main sections: Dashboard, Students, Staff, etc.

1.5 IDENTIFY GAPS VS. DESIRED DESIGN SYSTEM
----------------------------------------------------------------------

1.5.1 Compare current state to target design system concepts
  - Target concepts (from our earlier decisions):
      - Global design tokens (colors, spacing, typography).
      - JSON-driven configuration for:
          - Menus
          - Tables
          - Themes
      - Reusable components for:
          - Sidebar
          - Header
          - Page templates
          - Tables
      - Table behaviour:
          - Sticky header
          - Filter row under header
          - Global search on "Searchable" fields only
          - Icon-only inline action buttons
          - Skeleton loading rows
          - Empty state
          - Pagination
      - Icon system:
          - Fluent UI System Icons
          - Icon-only action buttons
      - Non-breaking, incremental refactor.

1.5.2 Create a gap analysis document
  - Create a new file:
      - /docs/gap-analysis-phase1.txt
  - Structure it as:
      - Section 1: Navigation
          - Current:
          - Target:
          - Gap:
      - Section 2: Sidebar & Layout
          - Current:
          - Target:
          - Gap:
      - Section 3: Tables
          - Current:
          - Target:
          - Gap:
      - Section 4: Styles & CSS
          - Current:
          - Target:
          - Gap:
      - Section 5: Configuration (JSON, etc.)
          - Current:
          - Target:
          - Gap:
      - Section 6: Components & Reusability
          - Current:
          - Target:
          - Gap:
  - Fill in each section based on:
      - /docs/structure-phase1.txt
      - Your knowledge of the target design system.

1.5.3 Classify gaps by impact and risk
  - In /docs/gap-analysis-phase1.txt, for each gap:
      - Add:
          - Impact: Low / Medium / High
          - Risk: Low / Medium / High
  - Example:
      - Tables:
          - Current: Each page defines its own CollectionView with custom logic.
          - Target: Reusable table component with JSON-driven column config.
          - Gap: No central table component; logic duplicated.
          - Impact: High (affects many pages).
          - Risk: Medium (refactor must be careful not to break existing tables).

1.5.4 Identify "safe to change early" vs "change later"
  - In /docs/gap-analysis-phase1.txt, add a final section:
      - "Refactor Priority:"
      - For each gap, assign:
          - Priority: Early / Mid / Late
  - Rules:
      - Early:
          - Changes that are low-risk and high-benefit.
          - Changes that do not break existing functionality.
          - Example: Adding new JSON config files, adding new styles without removing old ones.
      - Mid:
          - Changes that touch shared components but can be introduced gradually.
          - Example: Introducing a new table component and migrating one page at a time.
      - Late:
          - Changes that affect core navigation or layout.
          - Example: Replacing the sidebar structure or page templates.

1.6.1 Create a baseline summary document
  - Create:

  - Summarise:

      - Current component set.

  - Introduce CSS variable tokens, reusable .razor components, and JSON-driven configuration.
  - Ensure all new design system elements are additive and non-breaking.

Constraints:
  - Do NOT remove existing CSS.
  - Do NOT modify existing components.
  - All new design system elements must be isolated and namespaced.
  - All changes must be additive.

PHASE 3 — LAYOUT & NAVIGATION (BLAZOR VERSION)
Goal:
  - Establish a unified, reusable, scalable layout and navigation system for the Absence MAUI Blazor Hybrid app.
  - Introduce a new sidebar, header, breadcrumb, and page template system using .razor components.
  - Ensure all new layout elements are additive and do not break existing pages.
  - Prepare for gradual migration of existing pages to the new layout system.

Constraints:
  - Do NOT modify or remove existing Blazor layouts.
  - Do NOT break existing navigation or routes.
  - All new layout components must be introduced in parallel.
  - All new components must be namespaced and isolated.
  - All changes must be additive.

----------------------------------------------------------------------
3.1 CREATE NEW LAYOUT COMPONENTS
----------------------------------------------------------------------

3.1.1 Create folder for layout components
  - Path: /Components/LayoutV2/
  - Purpose:
      - Store new layout components without touching existing ones.

3.1.2 Create SidebarV2.razor
  - File: /Components/LayoutV2/SidebarV2.razor
  - Responsibilities:
      - Render menu items from menu.json (Phase 2).
      - Support nested menu groups.
      - Support expand/collapse.
      - Highlight active route.
  - Structure:
      - <nav> element
      - <ul> for top-level items
      - <li> for submenus
  - Must use CSS variables for colors and spacing.

3.1.3 Create SidebarV2.razor.css
  - File: /Components/LayoutV2/SidebarV2.razor.css
  - Responsibilities:
      - Style sidebar using tokens:
          --sidebar-bg
          --sidebar-text
          --sidebar-active-bg
          --sidebar-active-text
      - Support collapsed mode (optional).

3.1.4 Create HeaderV2.razor
  - File: /Components/LayoutV2/HeaderV2.razor
  - Responsibilities:
      - Display page title
      - Display optional subtitle
      - Display right-aligned action buttons
      - Display branding overrides (Phase 8)

3.1.5 Create HeaderV2.razor.css
  - Style header using:
      --header-bg
      --header-text
      --space-*

3.1.6 Create BreadcrumbV2.razor
  - File: /Components/LayoutV2/BreadcrumbV2.razor
  - Responsibilities:
      - Render breadcrumb trail
      - Accept list of breadcrumb items via parameter

3.1.7 Create BreadcrumbV2.razor.css
  - Style breadcrumb using:
      --breadcrumb-text
      --breadcrumb-separator

----------------------------------------------------------------------
3.2 CREATE NEW PAGE TEMPLATE SYSTEM
----------------------------------------------------------------------

3.2.1 Create folder for templates
  - Path: /Components/PageTemplatesV2/

3.2.2 Create PageTemplateV2.razor
  - File: /Components/PageTemplatesV2/PageTemplateV2.razor
  - Responsibilities:
      - Provide a two-column layout:
          Left: SidebarV2
          Right: HeaderV2 + BreadcrumbV2 + Body
      - Use CSS Grid or Flexbox
  - Parameters:
      - Title
      - Subtitle
      - BreadcrumbItems (List<string>)
      - ChildContent (RenderFragment)

3.2.3 Create PageTemplateV2.razor.css
  - Style layout using:
      --sidebar-width
      --header-height
      --content-padding

3.2.4 Create StandardPageV2.razor
  - File: /Components/PageTemplatesV2/StandardPageV2.razor
  - Responsibilities:
      - Wrap PageTemplateV2
      - Provide default layout for list pages, dashboards, settings pages

3.2.5 Create DetailPageV2.razor
  - File: /Components/PageTemplatesV2/DetailPageV2.razor
  - Responsibilities:
      - Wrap PageTemplateV2
      - Provide section-based detail layout
      - Support optional scroll-spy (Phase 3.4)

3.2.6 Create FormPageV2.razor
  - File: /Components/PageTemplatesV2/FormPageV2.razor
  - Responsibilities:
      - Wrap PageTemplateV2
      - Provide form layout with:
          - Form sections
          - Validation summary
          - Save/Cancel buttons

----------------------------------------------------------------------
3.3 NAVIGATION & ROUTING INTEGRATION
