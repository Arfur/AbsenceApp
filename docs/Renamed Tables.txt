NOTE: Group 2B (Users & Permissions Domain)

This domain contains all authentication and authorisation tables.
- Users represents login accounts (not staff identity).
- UserProfiles extends Users with additional profile information.
- Roles defines system-level permissions.
- RoleTypes categorises roles (e.g., Admin, Teacher).
- UserRole maps users to one or more roles.

This domain is intentionally separate from the Staff domain.
A staff member may or may not have a user account.
A user account may or may not belong to a staff member.

NOTE: GlobalConfig Table (Purpose and Future Review)

The GlobalConfig table currently stores two system-level switches:
- EnableGlobalSettings
- EnableRoleBasedNavigation

These switches act as master overrides for major system behaviour.
They are not linked to any other tables and are only read by the application.

EnableGlobalSettings:
- Controls whether the Global Settings module is active.
- Even though only Super-Admin users can access Global Settings, this flag can hide or disable the entire module at system level.

EnableRoleBasedNavigation:
- Controls whether role-based menu filtering and feature permissions are enforced.
- If set to false, all users see all menus and features (permissions bypassed).
- This is effectively a development/debugging switch and should never be disabled in production.

Current Assessment:
- The table adds an extra layer of complexity that may not be necessary.
- Both settings duplicate logic already enforced by roles and permissions.
- The table may be a legacy development feature rather than a required production component.

Action:
- Leave the table in place for now.
- Reassess during the refactor/cleanup phase.
- Consider removing or repurposing the table once the full system architecture is stabilised.
